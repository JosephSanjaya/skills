# Version Catalog Integration & Bootstrapping

## The Bootstrapping Paradox (Issue #15383)

Gradle parses `libs.versions.toml` and generates type-safe accessor classes (like `LibrariesForLibs`) during root project initialization. 
However, included builds (like `build-logic`) compile *prior* to root project evaluation.
* **Failure:** Trying to call `libs.libraries.retrofit` directly inside a precompiled script plugin results in compile-time `Unresolved reference: libs`.
* **Failure:** Applying plugin alias via `alias(libs.plugins.kotlin)` inside custom plugins plugins block fails because dynamic catalog getters do not exist on classpaths during convention compile phase.

---

## 4 Integration Workarounds

| Workaround | Autocomplete | Cache Performance | Maintenance | Coupling |
|---|---|---|---|---|
| **1. Classpath + Literal IDs** | No | **Excellent** | Moderate | Low |
| **2. VersionCatalogs API** | No | **Excellent** | Low | High |
| **3. BuildConfig Gen** | Yes | **Poor** | Low | Low |
| **4. Plugin-Owned (#36337)** | Yes | **Excellent** | Low | **Zero** |

### 1. Classpath Injection & Literal Strings (Recommended)
Declare catalog in `build-logic/settings.gradle.kts`:
```kotlin
dependencyResolutionManagement {
    versionCatalogs {
        create("libs") {
            from(files("../gradle/libs.versions.toml"))
        }
    }
}
```
Inject dependencies directly into the `build-logic/build.gradle.kts` compilation scope:
```kotlin
dependencies {
    implementation(libs.kotlin.gradle.plugin)
    implementation(libs.android.gradle.plugin)
}
```
Apply target plugins via string literals inside precompiled script:
```kotlin
// build-logic/src/main/kotlin/kotlin-conventions.gradle.kts
plugins {
    id("org.jetbrains.kotlin.jvm") // Version resolved from build-logic compilation classpath
}
```

### 2. Programmatic String-y API (`VersionCatalogsExtension`)
Safely query catalog alias strings dynamically during project configuration phase (bypasses code generation):
```kotlin
// build-logic/src/main/kotlin/library-conventions.gradle.kts
val catalogs = project.extensions.getByType<VersionCatalogsExtension>()
val libs = catalogs.named("libs")

dependencies {
    implementation(libs.findLibrary("dagger").get().get())
}
```
*Note:* IDE cannot perform usage detection on TOML variable access. 

### 3. BuildConfig Constant Generation (Avoid if possible)
Use `com.github.gmazzo.buildconfig` to parse `libs.versions.toml` at compile-time and output hardcoded constants.
```kotlin
// build-logic/convention/build.gradle.kts
plugins {
    `kotlin-dsl`
    id("com.github.gmazzo.buildconfig") version "5.3.0"
}
buildConfig {
    buildConfigField("String", "RETROFIT_VERSION", "\"${libs.versions.retrofit.get()}\"")
}
```
*Warning:* Bumping any version in the TOML file alters generated bytecode, invalidating `build-logic` compile classpath and resetting up-to-date state across the entire module graph.

### 4. Plugin-Owned TOML Catalogs (Gradle 9.x / Issue #36337)
Convention plugins declare and package their own isolated TOML catalogs.
* Target projects consume pre-bundled catalogs automatically.
* Zero configuration leakage, zero namespace conflicts, fully isolated dependency composition.
